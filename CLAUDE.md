## Architecture

**State Management:** Provider (`ChangeNotifier` / `ChangeNotifierProvider`). ViewModels own all mutable state; widgets call `context.watch<VM>()` or `context.read<VM>()`.

**Local Storage:** Hive for all offline-first data. All Hive boxes are opened in `main()` before `runApp`. Models that persist to Hive have a `.g.dart` file generated by `build_runner`. After editing any `@HiveType` class, regenerate adapters.

**Hive type IDs in use:**
| typeId | Class |
|--------|-------|
| 1 | SetType |
| 2 | PerformedSet |
| 3 | ExerciseLog |
| 4 | WorkoutHistoryEntry |
| 5 | PlannedSet |
| 50 | DurationAdapter (manual) |
| 101 | WorkoutTemplateModel |
| + | MeasurementEntry, MeasureProfile, MacroProfile, ExerciseNote (see their model files) |

**Backend:** PocketBase, accessed through a singleton `PB.I` (`lib/core/pb.dart`). The base URL is hardcoded to `http://10.0.2.2:8090` (Android emulator localhost). Auth token is persisted to `flutter_secure_storage` under key `pb_token`. Call `PB.I.bootstrapAuth()` at startup to restore sessions.

**Online/Offline mode:** `AppManager` (Provider) holds a `SessionMode` enum. The `Account` tab only appears in the bottom nav when `AppManager.isOnline == true`. Offline users have full access to templates, session tracking, history, and measurements without an account.

## Key Patterns

**Repository pattern:** Every Hive feature has an abstract `XRepository` interface and a `HiveXRepository` implementation. ViewModels depend on the interface, making the concrete store swappable (e.g. replace Hive with an API repo for online sync).

**Session flow:** `TemplatesPage` → `ViewTemplatePage` → `StartSessionFlow` (navigation helper in `templates/navigation/`) pushes `StartSessionPage`, which creates a scoped `WorkoutSessionViewModel` via `ChangeNotifierProvider`. On finish, the entry is saved through `HistoryViewModel.save()`.

**PR detection:** Happens inside `WorkoutSessionViewModel.markPlannedSetDone()` using `WorkoutSessionPrService`. PRs are stored separately in `prEventsBox`.

**Streak:** `StreakCalculator.compute()` in `lib/home/account/model/streakCalculator.dart` is pure (takes an `Iterable<DateTime>`) and runs on every Hive change inside `HistoryViewModel._recomputeDerived()`. `StreakSyncService` is wired in `main.dart` but sync is currently commented out (see `HistoryViewModel._maybeSync()`).

**Exercise catalog:** Exercises are a hardcoded static list in `exerciesesList.dart` (note the typo). They are referenced by integer `id`. The category icons live in `assets/workout_category/`.

## Adding a New Hive Model

1. Define the class with `@HiveType(typeId: N)` — pick a typeId not already in use.
2. Run `flutter pub run build_runner build --delete-conflicting-outputs`.
3. Register the adapter in `main()` with `Hive.registerAdapter(YourModelAdapter())`.
4. Open the box with `await Hive.openBox<YourModel>('yourBoxName')`.

## Backend Readiness Notes

All major data models (`WorkoutHistoryEntry`, `WorkoutTemplateModel`, `PerformedSet`, `ExerciseLog`, `PlannedSet`) already have `toJson()` / `fromJson()` methods. The repository abstraction layer is the intended API swap point — create an `ApiXRepository` that wraps PocketBase calls and inject it in place of the Hive implementation.

The `StreakSyncService` in `lib/home/account/model/streakSyncService.dart` is the stub for syncing streak data to PocketBase. Friends and social features already have service/viewmodel stubs in `lib/home/friends/`.
